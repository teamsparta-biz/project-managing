# 주강사 긴급구인 DM 스킬

`/urgent-main-instructor-recruit` 실행 시, **급하게 대체 인력이 필요한 특정 일자**에 가용 가능한 주강사를 ax-hub에서 **날짜별로** 조회해 사용자에게 먼저 보여주고, 사용자가 그중 **선별한 인원에게만** 개별 Slack DM으로 참여 가능 여부를 문의합니다.

> **핵심 원칙**: 교육 전체 일정을 한 사람이 다 커버할 수 있는지를 보는 게 아닙니다. **각 날짜마다 그날 가용한 사람이 누구인지**를 따로 보여줍니다. 긴급 구인은 보통 교육 전체 회차 중 "이 날짜 하나(또는 며칠)"에만 사람이 비어서 발생하기 때문입니다.

기존 `recruit-share`(채널에 공개 구인 글 게시)와 달리, 이 스킬은 **특정 인원을 지목해 1:1 DM**으로 문의하는 용도입니다. 기술튜터 대상은 [[urgent-tech-tutor-recruit]] 참고 — 이 스킬은 동일한 흐름을 주강사(role='main')에 적용한 버전입니다.

---

## 전체 흐름 요약

```
[1단계] 인자 파싱 (기업명 / 교육명 / 장소 / 전체일정 / 시간 / 담당자)
    ↓
[2단계] 긴급하게 확인할 "특정 일자" 확정 (인자에 없으면 질문)
    ↓
[3단계] ax-hub에서 해당 일자 각각의 가용 주강사를 날짜별로 조회 → 날짜별 표로 제시
    ↓
[4단계] 사용자가 DM 보낼 인원을 선별해 알려줌 (필수 대기 — 자동 선별 금지)
    ↓
[5단계] 선택 인원의 slack_id 확인 → 개별 DM 초안 미리보기
    ↓
[6단계] 사용자 확인 후 각자에게 개별 Slack DM 전송
```

---

## 1단계: 인자 파싱

명령 인자에서 아래 항목을 추출합니다. (웹앱 버튼은 이 형식으로 명령을 복사하며, 이때 `일정`에는 교육 전체 회차 날짜가 담깁니다.)

| 항목 | 설명 | 필수 여부 |
|------|------|----------|
| 기업명 | 교육 발주 기업 | 권장 |
| 교육명 | 교육 과정명 | 필수 (DM 본문에 사용) |
| 장소 | 교육 장소 | 필수 |
| 일정 | 콤마로 구분된 교육 전체 회차 날짜 목록 (참고용 — 2단계에서 "어느 날짜가 급한지" 물을 때 후보로 제시) | 권장 |
| 확인일자 | 콤마로 구분된, **가용 인원을 실제로 확인할 날짜**(급하게 대체가 필요한 날) | 없으면 2단계에서 질문 |
| 시간 | `HH:MM~HH:MM` | 권장 (없으면 시간 문구 생략) |
| 담당자 | 발신자 이름(웹앱 localStorage `training_owner` 값) | 필수 — 없으면 사용자에게 질문 |

누락된 필수 항목은 한 번에 모아서 질문합니다.

---

## 2단계: 확인할 특정 일자 확정

`확인일자` 인자가 이미 있으면 그대로 사용하고 이 단계를 건너뜁니다.

없으면, `일정`(전체 회차 날짜)이 있는 경우 그 목록을 보여주며 다음과 같이 질문합니다:

> **어느 날짜에 대체 주강사가 급하게 필요하신가요?** (한 개 또는 여러 개 선택 가능)
> {일정 목록 나열}

`일정`도 없으면 날짜를 직접 물어봅니다. **사용자가 지정한 날짜만** 3단계 조회 대상이 됩니다 — 전체 회차 날짜를 임의로 다 조회하지 않습니다.

---

## 3단계: 가용 주강사 조회 — 확인일자 "교집합" 우선

**사용자가 원하는 것은 지정한 확인일자 전체에 모두 가능한 사람을 가장 먼저 보는 것입니다.** 날짜별로 따로따로만 보여주는 것이 아니라, 교집합(전체 가능)을 최우선으로 제시합니다.

2단계에서 확정된 "확인일자" 목록으로 아래 SQL을 실행합니다. `ARRAY[...]` 안에는 확인일자만 채웁니다(전체 회차 날짜 전부가 아님). 확인일자가 1개든 여러 개든 동일한 쿼리로 처리됩니다.

```sql
SELECT i.id, i.name, i.phone, i.slack_id,
       array_agg(a.date ORDER BY a.date) FILTER (WHERE a.status = 'available') AS available_dates
FROM instructors i
JOIN instructor_segments s ON s.instructor_id = i.id
JOIN availability a ON a.instructor_id = i.id
WHERE s.role = 'main'
  AND i.is_active = true
  AND a.date = ANY(ARRAY['2026-07-21','2026-07-22']::date[])
  AND a.status = 'available'
  AND NOT EXISTS (
    SELECT 1 FROM assignments asg
    JOIN course_sessions cs ON cs.id = asg.course_session_id
    WHERE asg.instructor_id = i.id AND cs.date = a.date
  )
GROUP BY i.id, i.name, i.phone, i.slack_id
ORDER BY array_length(array_agg(a.date) FILTER (WHERE a.status = 'available'), 1) DESC, i.name;
```

> **중요**: `availability.status='available'`은 자기 신고 가용 상태일 뿐, 이미 다른 교육에 배정된 경우를 반영하지 않습니다. 반드시 `assignments`(+`course_sessions`)를 조인해 **그 날짜에 이미 배정이 있는 사람은 제외**해야 합니다. 이 조건을 빼면 이미 다른 교육에 배정된 사람이 "가능"으로 잘못 표시됩니다.

- `available_dates`의 길이가 확인일자 개수와 같은 사람 = **전체 가능** (교집합). 이 목록을 가장 먼저, 가장 눈에 띄게 보여줍니다.
- 일부만 일치하는 사람은 **일부 가능**으로 별도 구분해 보조적으로 보여주고, 어느 날짜만 가능한지 표시합니다.
- 결과 형식:

```
{교육명} 주강사 가용 현황 ({확인일자 목록})

전체 가능 (N명) — {확인일자 전부} 모두 가능
- {이름1}, {이름2}, ...

일부만 가능
- {이름}: {가능 날짜}만 가능
```

- 확인일자가 1개뿐이면 "전체 가능"과 "일부 가능" 구분 없이 그 날짜의 가용 인원만 나열합니다.
- 전체 가능 인원이 0명이면 그 사실을 먼저 알리고, 일부 가능 인원 중 조합으로 커버할 수 있는지 참고용으로 보여줍니다.

## 4단계: 사용자 선별 대기

**절대 자동으로 인원을 선택하지 않습니다.** 표를 보여준 뒤 아래처럼 질문하고 사용자 응답을 기다립니다:

> **이 중 DM을 보낼 인원을 알려주세요.** (어느 날짜에 대한 요청인지도 함께 알려주시면 정확합니다)

사용자가 이름을 알려주면(예: "7/15는 조효은, 7/16은 김성진") 해당 인원 + 해당 날짜만 다음 단계로 진행합니다. 한 사람이 여러 날짜에 배정될 수도 있습니다.

---

## 5단계: DM 초안 미리보기

선택된 인원의 `slack_id`를 3단계 조회 결과(또는 추가 조회)에서 확인합니다.

- `slack_id`가 없는 인원은 "슬랙 ID 미등록으로 자동 발송 불가"로 안내하고 DM 대상에서 제외합니다.

각 인원에게 아래 템플릿으로 **이름·날짜만 다르게** DM을 작성합니다. 날짜는 그 사람에게 요청하는 **해당 날짜(들)만** 명시합니다 (교육 전체 일정이 아님):

```
**[{교육명} 주강사 참여 가능 여부 문의]**

안녕하세요, {이름}님. 팀스파르타 {담당자} 매니저입니다.

다음 일정의 주강사 참여가 가능하신지 확인 차 연락드립니다.

- 교육명: {교육명}
- 요청 일자: {해당 인원에게 요청하는 날짜(들) — "YYYY-MM-DD(요일)"}
- 시간: {시간} (시간 인자 없으면 이 줄 생략)
- 장소: {장소}

참여 가능하신지 회신 부탁드립니다. 감사합니다.
```

전송 대상자 목록과 메시지 내용을 미리보기로 출력하고 아래 문구로 마무리합니다:

> **위 내용으로 {N}명에게 개별 DM을 전송할까요? 수정할 내용이 있으면 말씀해 주세요.**

---

## 6단계: 개별 Slack DM 전송

사용자가 전송을 확인하면(예: "전송해줘", "네", "검수완료"):

1. 대상자 각각에게 `slack_send_message`로 `channel_id`에 해당 인원의 `slack_id`를 지정해 개별 발송합니다. (채널이 아닌 1:1 DM — user_id를 channel_id로 사용)
2. 전송 완료 후 아래처럼 결과를 정리해 보여줍니다:

> **✅ {N}명에게 개별 DM을 전송했습니다: {이름(요청일자) 목록}**
> (slack_id 미등록으로 제외된 인원이 있으면 함께 안내)

전송 실패한 인원이 있으면 오류 내용과 함께 안내하고 재시도 여부를 묻습니다.

## 사용 예시

```
/urgent-main-instructor-recruit 기업명:삼성바이오로직스 교육명:AI Frontier 교육 장소:인천광역시 연수구 송도바이오대로 300 일정:2026-07-06,...,2026-07-31 시간:13:00~17:00 담당자:송찬호
→ "어느 날짜가 급하신가요?" 질문 → 사용자가 "7/21, 7/22" 선택
→ ax-hub에서 7/21·7/22 교집합 조회 → "전체 가능(양일 모두)" 목록 먼저 출력, 일부 가능은 보조로 표시
→ 사용자가 "박현일, 최예찬" 선택 → DM 미리보기(7/21, 7/22만 언급) → 확인 → 2명에게 개별 전송
```

## 주의사항

- `mcp__ax-hub__query_sql`은 SELECT만 허용됩니다 (read-only).
- **확인일자로 지정된 날짜만** 조회합니다 — 전체 교육 회차 날짜를 다 조회해서 "모든 날짜를 커버할 수 있는 사람"을 찾는 방식이 아닙니다.
- 단, 확인일자가 여러 개(예: 2개 이상)면 그 확인일자들 **모두 가능한 사람(교집합)을 항상 최우선으로** 보여줍니다. 날짜별로 뿔뿔이 흩어서만 보여주지 않습니다.
- `availability.status='available'`만 보고 판단하지 않습니다. 이미 그 날짜에 다른 교육 `assignments`가 있는 사람은 실제로는 불가능하므로 반드시 제외합니다 ([[feedback_availability_excludes_assigned]] 참고).
- 3단계 조회 결과를 사용자에게 보여주기 전까지 DM을 발송하지 않습니다.
- 인원 선별은 반드시 사용자가 직접 지정합니다 — 스킬이 임의로 인원을 자동 선택해 보내지 않습니다.
- `slack_id`가 없는 인원은 발송 대상에서 제외하고 반드시 안내합니다.
- DM은 채널이 아닌 1:1 대화로 전송합니다 (`channel_id`에 대상자의 Slack user_id 사용).
- DM 본문에는 그 사람에게 실제로 요청하는 날짜만 적습니다 — 교육 전체 일정을 나열하지 않습니다.
- 사용자가 전송을 확인하기 전까지 미리보기 직후 자동 전송 금지.
